## Context

当前项目的默认 agent composition 位于 `src/agent.py`，默认产品语义仍是 `SmartIPO` 的 “美股 IPO / 财报 / 估值研究助手”。TUI 入口位于 `src/tui/app.py`，其当前启动行为只负责挂载界面、刷新视图与聚焦输入框，不会主动注入一条 assistant 开场消息；因此“agent 先于用户说话”在现状下不是 prompt 问题，而是界面流程问题。

现有数据能力以 FMP 工具集为主，已经覆盖 IPO 日历、招股书 / prospectus、SEC filings、基础估值指标、财报、owner earnings、历史价格、历史市值、公司筛选与部分治理/内部人数据。这足以支撑“美股专业级估值分析”与“打新分析”的第一阶段产品定义，但并不足以支撑对期货、期权、广义衍生品、加密资产做等强度承诺，因此新定义必须把“现有可验证数据覆盖边界”写清楚。

这次变更同时具备三类交叉影响：

- 品牌重定义：`SmartIPO` -> `OpenBuffett`
- 研究主链重定义：`IPO-first` -> `valuation-first`
- 交互节奏重定义：被动等待用户 -> 启动主动欢迎 + 分析后主动追问报告

它跨越默认 prompt、TUI 首屏行为、状态文案、README、包元数据与测试断言，属于典型的跨模块产品重定义，不适合只做零散字符串替换。

## Goals / Non-Goals

**Goals:**

- 让默认 agent 的产品身份、品牌露出、能力优先级与系统提示词统一收敛到 `OpenBuffett`。
- 把“专业级估值分析”固化为默认研究主链，并要求输出结论先行、事实分级、可比公司对照、多周期估值视角与风险揭示；其中多周期默认采用 `1个月 / 6个月 / 1年 / 3年 / 5年`，并要求短期视角显式覆盖货币/经济政策、地缘政治、广泛市场情绪与资金流向。
- 把信息面获取约束固化为硬规则：涉及宏观、政策、地缘、市场情绪、资金流、新闻事件与 IPO 进度时，必须联网获取最新信息，区分来源可信度，识别情绪化叙事，并保留最小审计日志。
- 为公司名 / 模糊名 / 疑似笔误输入增加前置 ticker 解析与确认门禁，避免错误标的直接进入估值分析。
- 把“打新分析”明确约束为估值主链之上的延伸流程，并加上未上市/待申购的适用门禁。
- 让 TUI 启动后立即出现一条 agent 主动开场消息，并把欢迎话术维护为独立常量，而不是散落在界面代码里。
- 让 agent 在完整研究结尾主动询问用户，是否要在当前工作路径生成中文 Markdown 研究报告。
- 在不改历史归档文件的前提下，清理当前运行面与现行项目表面的旧品牌标识。

**Non-Goals:**

- 不重命名历史 OpenSpec 归档文件、历史 proposal/design/spec 文档或历史变更记录中的旧品牌。
- 不在本轮引入新的数据供应商、数据库、缓存层、研究评分器、多 agent 编排器或专门的报告持久化子系统。
- 不在本轮把期货、期权、广义衍生品或加密数据扩展成完整研究栈；若当前工具不能验证，系统只需诚实暴露边界。
- 不在本轮改动仓库工作目录名或大规模迁移 Python 包结构；品牌重定义先落在产品表面、元数据与行为合同。

## Decisions

### 决策 1：保持 `src/agent.py` 为单一 composition 入口，并在其中集中维护品牌、开场与默认 prompt 常量

决策：继续把默认模型配置、system prompt、品牌常量、欢迎话术常量与默认工具集合集中在 `src/agent.py`，并由 TUI 只消费这些导出常量，而不是在 TUI 内部重复维护产品文案。

理由：

- 当前项目已经把 `src/agent.py` 作为唯一 composition 层，这一层天然适合承载“产品身份定义”。
- 欢迎话术若散落在 TUI，会导致品牌、能力介绍与 system prompt 逐步漂移。
- 把品牌与 prompt 常量集中到单一文件，审查成本最低，也最利于未来再次重定义。

备选方案：

- 新增独立 `src/prompt/` 或 `src/brand/` 模块承载常量。拒绝，因本轮真实改动规模还不足以支付新目录与新抽象成本。
- 继续把欢迎文案写死在 TUI。拒绝，因为这与“开场自介解耦于 TUI 维护”的用户要求直接冲突。

### 决策 2：把“估值分析主链”主要实现为高密度结构化 system prompt 合同，而不是额外新建状态机或研究编排器

决策：估值主链、打新门禁、事实分级、比较公司要求、结尾报告追问等，优先通过结构化 system prompt 约束为默认行为合同；仅对“启动即开场”这类无法由 prompt 单独完成的行为做最小 TUI 配合。

理由：

- 这次重定义的核心是产品职责与研究纪律变化，而不是新增复杂 runtime。
- 当前 EasyHarness + FMP 工具面已经足够支撑 agent 在单轮里完成多步研究；过早引入编排器只会把“prompt 重定义”误做成“框架扩建”。
- 用户明确强调要在 system prompt 中高密度注入投资研究方法，而非把知识点转移到一层额外 orchestration 中。
- 用户进一步明确指出，原本的 `1/5/10/15 年` 框架不贴近当前业务，应改成更可验证、更贴近市场实务的 `1个月 / 6个月 / 1年 / 3年 / 5年`，并把短期宏观与资金面因素设为硬约束。
- 用户进一步强调，信息面绝不能依赖静态知识或单一不可靠媒体；必须联网取最新、分级核信、识别情绪化判断，并保持完全去情绪化、可审计的表达。

补充约束：

- `1个月 / 6个月` 的短期判断，不得只写“情绪偏乐观/偏谨慎”之类空话；必须尽量落到可观察代理变量，例如利率路径、联储口径、通胀/就业数据、收益率曲线、美元、油价、VIX、信用利差、ETF/板块资金流、相对强弱与仓位拥挤度。
- `1年` 视角同时看基本面兑现、估值回归与市场制度环境。
- `3年 / 5年` 视角回到商业质量、财务质量、资本回报与竞争格局主导。
- 凡涉信息面、政策、地缘、市场情绪、资金流、新闻事件与 IPO 进度，必须联网获取最新信息并标注分析日期。
- 来源可信度至少区分四级：A=官方一级源；B=结构化金融数据源；C=主流媒体/行业媒体；D=社媒、论坛、传闻、二手转述。关键结论须优先由 A/B 支撑。
- 输出中必须显式区分“已核事实”“可观察代理变量”“媒体叙事”“情绪化判断/未经证实传闻”，并保留精简版来源、时间戳、可信等级与冲突点说明。

备选方案：

- 为 ticker 路由、估值分析、打新分析单独建设显式工作流状态机。拒绝，因实现重、验证面广，且当前阶段缺乏足够复杂度证据。
- 只改 TUI 或只改工具，不改 prompt。拒绝，因为研究纪律与输出契约本质上属于 prompt 合同。

### 决策 3：将“启动即开场”设计为 TUI 注入的首条 assistant 历史消息，而不是触发一次空 prompt 轮次

决策：在 `on_mount()` 阶段直接把默认欢迎话术作为 assistant 消息写入时间线，而不是发起一轮空输入或伪输入给 agent runtime。

理由：

- 空 prompt 轮次会污染真实会话历史，也可能触发不稳定的模型行为、无意义工具调用或额外 token 消耗。
- 启动欢迎语本质上是产品固定文案，不需要模型参与生成。
- 直接注入历史消息最可控，也最容易测试。

备选方案：

- 启动时自动向 agent 发送一条隐藏 prompt 生成欢迎语。拒绝，因为欢迎语应稳定、可审核、与模型随机性解耦。
- 保持现状，仅依赖用户第一句后由 agent 再介绍自己。拒绝，因为这无法满足“agent 先于用户说话”的产品要求。

### 决策 4：把“模糊公司名 -> ticker 解析 -> 用户确认”定义为硬门禁，确认前禁止进入正式估值结论

决策：当用户未直接提供明确股票代码时，agent 必须先用可验证搜索/数据源推断候选 ticker，并在用户确认前停在候选阶段；不得一边猜代码、一边继续生成正式估值或打新结论。

理由：

- 错标的下的深入估值研究是高成本错误，越晚发现越难纠偏。
- 用户已经明确把这一步定义为强要求，而非可选礼貌动作。
- 在投资研究场景中，标的确认属于先验门禁，不应被“为了流畅”跳过。

备选方案：

- 当置信度高时默认继续分析。拒绝，因为这仍然违背“主动向用户确认，在用户确认后才能继续”的硬要求。
- 把 ticker 解析做成独立命令。拒绝，因为用户希望这是默认对话能力，而非额外操作负担。

### 决策 5：将“打新分析”建模为估值主链的受限分支，而不是与估值分析并列的独立入口

决策：打新分析必须建立在估值分析已完成或可复用的前提上，并先通过“是否仍属于未上市且可申购/待申购阶段”的门禁；若目标已上市，则自动退回到常规估值/二级市场分析路径。

理由：

- 用户已明确指出打新分析是能力 1 的延伸，而非并列主干。
- 招股书、发行结构、冻结解冻、锁定期与配售结构都必须站在公司估值和经营质量判断之上才有意义。
- 这能避免把所有 IPO 问题都粗暴拉进“打不打新”的狭窄框架。

备选方案：

- 允许直接对任何上市公司做“打新回顾式分析”。拒绝，因为会混淆一级申购决策与二级市场交易研究。

### 决策 6：对第 4 类“其他金融产品”能力采取保守承诺，只承诺现有工具可验证覆盖的部分

决策：在新 prompt、文档与欢迎话术中，只承诺股票、ETF 及当前工具面可覆盖的历史价格 / 市值研究；对暂未稳定覆盖的资产类型必须允许 agent 明确说“当前数据面不足，无法严谨分析”。

理由：

- 当前 FMP 工具已经能支撑股票/ETF 风格的时间序列与估值复盘，但不足以支撑完整跨资产研究栈。
- 保守承诺比“产品宣称很大、实际验证很弱”更符合用户对严谨性的要求。

备选方案：

- 直接把期货、期权、比特币等写成默认完整支持。拒绝，因为实现证据不足，属于高风险过承诺。

### 决策 7：研究报告生成先保持“询问 + 用户确认后再写入”的合同，不自动落盘

决策：agent 在完整分析结束时必须主动询问是否生成研究报告，但未获用户确认前不得自动写入当前工作路径。报告默认中文 Markdown，结构结论先行，并要求显式标注证据等级与未知项。

理由：

- 用户要求是“主动问要不要生成”，不是“自动生成”。
- 自动落盘会在多轮分析、探索阶段或临时问答中制造大量噪声文件。
- “先问再写”与当前 `fileglide` 写入能力兼容，不需要额外引入报告系统。

备选方案：

- 每轮分析结束都自动写报告。拒绝，因为会显著提升工作区噪音与误写入风险。
- 只在 README 说明可手动要求生成，不主动问。拒绝，因为不符合用户要求的默认交互节奏。

## Risks / Trade-offs

- [Risk] system prompt 变得更长、更密，后续维护者更难随手改动。  
  → Mitigation：把 prompt 固化为少数清晰分段，集中在单一 composition 文件，并通过测试断言关键语义。

- [Risk] 加入短期宏观与资金面约束后，agent 更容易“装懂宏观”。  
  → Mitigation：在 prompt 中明确要求引用可观察代理变量；若证据不足，必须承认短期判断置信度受限，而不是照样给出斩钉截铁的短期结论。

- [Risk] 加入“联网最新信息”要求后，agent 可能被低可信或情绪化信息源带偏。  
  → Mitigation：把来源可信度分级、情绪化叙事降权与最小审计日志写成硬约束；关键结论优先回到 A/B 级来源。

- [Risk] 欢迎语由 TUI 注入而非模型生成，可能与后续 prompt 微调出现语义漂移。  
  → Mitigation：欢迎语常量与 system prompt 同处 `src/agent.py`，统一维护并增加回归测试。

- [Risk] ticker 门禁会让部分对话多一步确认，牺牲少量流畅度。  
  → Mitigation：仅在输入非明确 ticker 或存在歧义时触发；若用户已给标准代码，则直接进入分析。

- [Risk] 对跨资产能力的保守承诺看起来“不够大而全”。  
  → Mitigation：把可信边界说清楚，优先保证股票/ETF/IPO 主链专业度；等真实数据栈补齐后再扩展承诺。

- [Risk] 当前仓库目录仍名为 `SmartIPO`，与产品名 `OpenBuffett` 存在短期不一致。  
  → Mitigation：本轮先改产品表面、包元数据与对外文案；若未来需要再单独立项做仓库与分发层迁移。

## Migration Plan

1. 先重写默认 prompt 与导出常量，建立 `OpenBuffett` 的品牌和研究合同。
2. 在 TUI 注入首屏欢迎语，统一标题、placeholder、状态文案与系统消息品牌名。
3. 更新 README、包元数据与当前有效源码表面；明确历史归档文件不在迁移范围内。
4. 补充 / 重写测试，覆盖品牌替换、启动欢迎、prompt 关键语义、ticker 确认门禁与报告追问。
5. 若实现过程中发现“跨资产能力承诺”与当前工具面仍存在隐藏冲突，则优先缩窄 prompt 承诺，而不是临时补大型数据能力。

回滚策略：

- 若新 prompt 造成明显行为回归，可先回退 `src/agent.py` 的 prompt 与品牌常量改动，同时保留 TUI 技术性修复；
- 若欢迎语注入影响会话历史展示，可单独回退 TUI 开场逻辑，不影响估值主链重定义；
- 品牌替换应与行为变更分批提交，保证必要时可以分别回滚。

## Open Questions

- 是否要在本轮顺带把 `pyproject` 的分发名也改为 `openbuffett`。当前建议是改，因为它属于现行对外元数据，但这可能影响已有本地安装名称。
- 是否要在研究报告询问后同时提供一个固定建议文件名模式，例如 `openbuffett-report-<symbol>-<date>.md`。当前提案先不强制，只要求问询与默认中文结构。
- 对“其他金融产品”的边界表述是否需要进一步细化到“股票/ETF 明确支持、其他资产视数据覆盖而定”。当前设计已偏保守，但实现时仍需避免文案过度泛化。
