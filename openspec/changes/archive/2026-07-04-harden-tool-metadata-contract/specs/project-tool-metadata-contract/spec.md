## ADDED Requirements

### Requirement: 项目自有工具必须提供稳定且有区分度的元数据合同
系统 MUST 为项目自有工具提供稳定的元数据合同，至少覆盖 `purpose`、`when_to_use`、`parameters`、`returns` 与 `common_failures`。这些字段 MUST 能帮助模型区分相邻工具的适用边界，而不是只复述“获取某个数据”的表层动作。

#### Scenario: 两个相邻 FMP 工具同时存在
- **WHEN** 默认工具集中同时存在功能接近但边界不同的 FMP 工具，例如报价、历史价格、可比公司或筛选类工具
- **THEN** 它们的 `when_to_use` 描述 MUST 能明确区分适用场景
- **AND** 系统 MUST NOT 仅依赖统一模板拼接出几乎不可区分的使用说明

### Requirement: 项目自有工具参数说明必须体现高频输入与关键约束
系统 MUST 为项目自有工具提供可执行的参数说明。对于显式参数，说明 MUST 指出格式、默认语义或关键约束；对于 `extra_params` 这类结构化透传字段，说明 MUST 给出高频参数类别或代表性示例，而不是仅写“传入 JSON 对象”。

#### Scenario: 调用方查看筛选类或查询类工具的参数
- **WHEN** 调用方查看某个依赖结构化透传字段的工具参数说明
- **THEN** 系统 MUST 给出足以指导首次调用的代表性参数示例或参数类别
- **AND** 系统 MUST 明确显式参数与透传参数的优先级关系

### Requirement: 项目自有工具返回说明必须表达结果形状
系统 MUST 在 `returns` 中说明工具返回的大体形状和关键载荷类型，例如单对象、列表、时间序列或文本摘要，而不是只笼统描述“返回原始 JSON”。

#### Scenario: 调用方比较快照型工具与时间序列型工具
- **WHEN** 调用方查看 `quote`、`historical price`、`historical market cap` 或类似工具的返回说明
- **THEN** 系统 MUST 能体现这些工具在结果形状上的差异
- **AND** 系统 MUST 让调用方可以在调用前预判输出是否适合当前分析任务

### Requirement: 项目自有工具元数据必须与项目语言策略一致
系统 MUST 对项目自有工具采用一致的元数据语言策略，并与 SmartIPO 面向中文研究任务的默认交互方向保持一致。若外部工具来源不可控，系统 MUST 明确区分“项目自有文案标准”与“外部工具保留原样”的边界。

#### Scenario: 默认工具面同时包含项目自有工具与外部工具
- **WHEN** 默认 agent 装配项目自有工具和官方外部工具
- **THEN** 系统 MUST 保证项目自有工具文案风格内部一致
- **AND** 系统 MUST NOT 为了风格统一而重写第三方官方工具的实现来源
