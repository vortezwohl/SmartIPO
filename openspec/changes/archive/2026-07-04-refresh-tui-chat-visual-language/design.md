## Context

当前 `AgentWorkbenchApp` 虽然已经具备串行队列能力，但界面仍采用“一个 `Static` 里拼整段文本”的渲染方式：消息、工具状态、thinking 和排队文案都混在同一条时间线上。这种结构适合验证事件流是否工作，不适合表达你来我往的聊天体验，也很难把排队状态做成独立的中部居中泡泡区。

这次变更的目标不是修改底层 agent 或队列语义，而是重构 TUI 的展示层，让现有状态模型能投射到更自然的聊天工作台上。重点在于：
- 对话记录和排队状态分区；
- 从日志风文本转向轻量聊天气泡；
- 用明亮浅绿色主题统一界面气质。

## Goals / Non-Goals

**Goals:**
- 把待处理队列从 timeline 主消息区中拆出，放入输入框上方的独立排队托盘。
- 让对话区使用 `👨‍💻` / `🤖` 等角色标识，替代 `你:` / `AI:` 或方括号状态标签。
- 把整体配色调整为明亮浅绿色主调，并让主要消息文本保持白色。
- 更新输入区 placeholder 与相关状态文案，减少工具日志感。

**Non-Goals:**
- 不改变底层 FIFO 队列语义、会话重置语义或旧事件忽略逻辑。
- 不把整个 TUI 改造成复杂的自定义聊天组件系统。
- 不新增图像、字体依赖或外部主题包。

## Decisions

### 1. 增加独立 `queue-tray` 区域，而不是把排队项继续塞进 timeline

布局会从“状态栏 + timeline + input”升级为“状态栏 + timeline + queue tray + input”。排队中的消息只在 `queue-tray` 中显示，不再作为 `user` 时间线项渲染到对话记录里。

这样做的原因是排队消息和已发生对话是两类信息：
- timeline 负责“已经进入会话处理或已经产生结果的事实”；
- queue tray 负责“即将被处理的待办”。

如果继续混放，队列越长，聊天主视图越像待办列表。

### 2. 保留现有状态模型，重做渲染出口

不推翻 `_items`、`_pending_turns` 和 `active_turn` 这些状态，只改“怎么显示”。最小路径是：
- timeline 继续由本地状态驱动；
- queue tray 直接读取 `_pending_turns`；
- 渲染上从纯字符串拼接，升级为 Rich/Textual 现有 renderable 组合。

优先复用现有状态，而不是重新建立第二套队列展示模型，避免视觉改造把业务语义重新做一遍。

### 3. 聊天消息改用角色图标和轻量气泡风格

用户消息和助手消息的前缀改为图标，不再显示 `你:`、`EasyHarness:`、`AI:`。视觉上应更接近：
- `👨‍💻 hi`
- `🤖 hi im SmartIPO`

同时减少 `[]`、`|`、`:` 这类调试型符号的占比。对于非正文消息，如工具状态、thinking 或系统错误，也应尽量使用自然短句或轻标签，而不是大段方括号状态串。

### 4. 主题用浅绿色背景氛围 + 更深一层的绿色承载白字

用户要求“整体色调明亮一点、不要暗沉沉的、字体用白色”。直接把极浅绿底配纯白字会损失对比度，因此采用两层绿色：
- 外层背景与面板氛围使用明亮浅绿；
- 消息卡片、输入框和队列泡泡使用稍深一点但仍然明亮的绿色；
- 主要正文使用白色。

这样既满足白字和浅绿色方向，又不会牺牲可读性。

### 5. 排队顺序通过空间位置表达，避免重方括号顺位标签

queue tray 中不再渲染类似 `[排队中 #3]` 的字面标签。默认通过泡泡排列顺序表达谁先谁后；如果需要辅助顺位信息，也只用轻量的小数字或次级说明，不进入正文。

## Risks / Trade-offs

- [Risk] 当前 timeline 是字符串渲染，升级到更复杂的 renderable 后，测试需要从文本断言转向更关注区域文案和状态事实。 → Mitigation：保留必要的文本快照能力，但把断言重心放到“是否出现在正确区域”。
- [Risk] 过度追求聊天 UI，导致工具事件、失败信息变得不够可读。 → Mitigation：正文消息做得更自然，但诊断信息仍保留简洁明确的展示层。
- [Trade-off] 不引入完整自定义消息 widget 系统，意味着第一版气泡样式会偏轻量。 → 这是刻意控制复杂度；先把分区、主题和角色语言做对，再决定是否继续强化视觉细节。

## Migration Plan

1. 先改布局与 CSS，增加 queue tray，并替换 placeholder 与主题变量。
2. 再调整 timeline 和 queue tray 的渲染函数，确保排队消息移出主消息区。
3. 最后更新测试，验证视觉语义和区域边界。

## Open Questions

- tool / thinking / system 这些非聊天正文消息，最终是否也要进一步做成更柔和的卡片，而不是纯文本行。
- queue tray 中是否需要显示正在处理的当前消息的“输入回显泡泡”，还是只显示尚未开始的待处理消息。
