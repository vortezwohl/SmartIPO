## ADDED Requirements

### Requirement: SmartIPO TUI SHALL use English for all fixed UI elements
The SmartIPO TUI MUST render all fixed UI elements in English, including the Header title, status summary, queue tray label, input placeholder, local slash-command feedback, and timeline activity labels generated by the UI layer. Agent-generated content MAY remain in Chinese or any other language.

#### Scenario: Header and local controls use English labels
- **WHEN** the TUI renders its Header, status banner, queue tray, input placeholder, or local command feedback
- **THEN** those fixed UI elements MUST use English text
- **AND** the assistant reply body MUST NOT be forcibly translated by the UI layer

### Requirement: SmartIPO TUI SHALL brand the top Header as SmartIPO
The SmartIPO TUI MUST replace the default framework application title in the top Header with `SmartIPO` and MUST render that Header with a darker green brand background than the ordinary chat prefix accent color.

#### Scenario: Header shows SmartIPO instead of default framework title
- **WHEN** the TUI mounts and renders the top Header
- **THEN** the visible application title MUST be `SmartIPO`
- **AND** the Header background MUST use a darker green brand surface distinct from the lighter chat accent color

### Requirement: Thinking activity SHALL be a temporary pre-reply state
The SmartIPO TUI MUST render local waiting activity as `Thinking ...` using the format `duration · action`. This thinking entry MUST exist only as a temporary dynamic state before the assistant actually starts producing output.

#### Scenario: Waiting for first assistant output shows Thinking
- **WHEN** the user submits a prompt and no real assistant output has arrived yet
- **THEN** the timeline MUST show a temporary activity line in the format `duration · Thinking ...`

#### Scenario: Thinking disappears after assistant output begins
- **WHEN** the assistant starts or streams its first real output for the active turn
- **THEN** the temporary thinking entry MUST disappear from the visible timeline
- **AND** the timeline MUST continue with the assistant message history instead of preserving the completed thinking item

### Requirement: Activity timing SHALL use a unified separator format
The SmartIPO TUI MUST format activity timing as `<duration> · <action>` for timeline items that represent temporary or running activities, including thinking and tool activity main lines.

#### Scenario: Tool and thinking activities share one timing grammar
- **WHEN** the TUI renders a thinking line or a tool activity main line
- **THEN** the duration text MUST appear first
- **AND** it MUST be followed by ` · `
- **AND** the activity text MUST appear after that separator

### Requirement: Tool activity SHALL use a braced main line and English detail labels
The SmartIPO TUI MUST render the first line of a tool activity as `duration · { Tool <name> · <status> }`. Tool follow-up lines MUST use English fixed labels such as `Call`, `Summary`, `Result`, and `Error`.

#### Scenario: Running tool uses braced main line
- **WHEN** a tool activity starts running
- **THEN** the main tool line MUST be rendered in the format `duration · { Tool <name> · Running }`

#### Scenario: Completed tool keeps summary lines outside the braces
- **WHEN** a tool activity completes with preview or detail content
- **THEN** the first visible tool line MUST remain the braced main line
- **AND** any `Call`, `Summary`, or `Result` lines MUST be rendered as follow-up lines outside the braces

#### Scenario: Failed tool uses English error labels
- **WHEN** a tool activity fails
- **THEN** the main tool line MUST use an English failed status
- **AND** any visible error detail line MUST use the `Error` label
